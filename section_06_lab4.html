🎯 Code Challenge 1
Generate dim_date entirely using a SQL query — no source table needed. Use the Spark SQL sequence() function to generate one row per day from 2014-01-01 to 2017-12-31, then derive all calendar columns.
  • →Use spark.sql() with a multi-line SQL string
  • →Inner subquery: SELECT explode(sequence(date('2014-01-01'), date('2017-12-31'), interval 1 day)) AS d
  • →Outer SELECT: derive all 10 columns from d (see list below)
  • →date_key: integer in format YYYYMMDD — cast date_format(d,'yyyyMMdd') to INT
  • →is_weekend: dayofweek returns 1=Sun, 2=Mon... 7=Sat — weekend is 1 (Sun) or 7 (Sat)
  • →Write to Delta as dim_date, print row count (expected: 1,461)
💡 Hint — SQL skeleton
dim_date = spark.sql(""" SELECT CAST(date_format(d, '_____') AS INT) AS date_key, -- YYYYMMDD int d AS date, year(d) AS year, quarter(d) AS quarter, month(d) AS month_num, date_format(d, '_____') AS month_name, -- 'MMMM' = January date_format(d, '_____') AS month_short, -- 'MMM' = Jan (dayofweek(d) IN (_____, _____)) AS is_weekend, -- 1=Sun, 7=Sat date_format(d, '_____') AS year_month, -- 'yyyy-MM' concat('Q', quarter(d)) AS quarter_label FROM ( SELECT explode( sequence(date('_____'), date('_____'), interval 1 day) ) AS d ) """) dim_date.write.format("delta").mode("overwrite").saveAsTable("_____") print(f"dim_date : {dim_date.count()} rows") -- expected 1461
date_format patterns used: 'yyyyMMdd' → '20140103' (for date_key integer) 'MMMM' → 'January' (full month name) 'MMM' → 'Jan' (3-letter short name) 'yyyy-MM' → '2014-01' (for year_month label) dayofweek() in Spark SQL: 1 = Sunday, 2 = Monday, ... 6 = Friday, 7 = Saturday So is_weekend = dayofweek IN (1, 7)

Cell 3 — dim_date
# ── Load source tables ────────────────────────────────────────
s  = spark.table(SRC_SALES)
cn = spark.table(SRC_COUNTRIES)

# ── 1) dim_date — generated entirely by SQL sequence ──────────
dim_date = spark.sql("""
    SELECT
        CAST(date_format(d, 'yyyyMMdd') AS INT) AS date_key,
        d                                        AS date,
        year(d)                                  AS year,
        quarter(d)                               AS quarter,
        month(d)                                 AS month_num,
        date_format(d, 'MMMM')                  AS month_name,
        date_format(d, 'MMM')                    AS month_short,
        (dayofweek(d) IN (1, 7))                 AS is_weekend,
        date_format(d, 'yyyy-MM')               AS year_month,
        concat('Q', quarter(d))                  AS quarter_label
    FROM (
        SELECT explode(
            sequence(
                date('2014-01-01'),
                date('2017-12-31'),
                interval 1 day
            )
        ) AS d
    )
""")
dim_date.write.format("delta").mode("overwrite").saveAsTable("dim_date")
print(f"dim_date      : {dim_date.count()} rows")   # expected 1461
Why generate dim_date with SQL instead of from the data? If we only took dates from silver_sales, dim_date would only have rows for dates that appear in the data — no weekend rows without orders, no January 1st entries, etc. A complete, contiguous date table is a Power BI requirement for time intelligence (YoY, MTD, QTD) to work correctly. The SQL sequence generates every single day regardless of whether it has sales.

Why 1,461 rows? 2014-01-01 to 2017-12-31 = 4 years. 2016 is a leap year (366 days). So: 365 + 365 + 366 + 365 = 1,461.
Instructor Note
This is a great moment to explain why a continuous date table matters for Power BI time intelligence. Ask: "If dim_date only had dates with sales, what would DATESYTD() return on January 1st?" The answer: nothing — DATESYTD needs every date in the year to be present. Also demonstrate: dayofweek(date('2014-01-04')) = 7 → Saturday → is_weekend = true. Students can verify this directly in the SQL endpoint.

dim_customer = (s .select("customer_id", "customer_name", "_____") .dropDuplicates(["_____"]) .withColumn("customer_sk", F.row_number().over(Window.orderBy("_____"))) # order by natural key .withColumn("_ts", F.current_timestamp())) dim_customer.write.format("_____").mode("overwrite").saveAsTable("_____") print(f"dim_customer : {dim_customer.count()} rows") dim_product = (s .select("product_id", "_____", "_____", "_____") .dropDuplicates(["_____"]) .withColumn("product_sk", F.row_number().over(Window.orderBy("_____"))) .withColumn("_ts", F.current_timestamp())) dim_product.write.format("_____").mode("overwrite").saveAsTable("_____") print(f"dim_product : {dim_product.count()} rows")
Why order by the natural key? Window.orderBy("customer_id") ensures the SK assignments are DETERMINISTIC. Every time you run this notebook, CG-12520 (Claire Gute) gets the same customer_sk. If you use Window.orderBy(F.rand()), the SK changes on every run — breaking any existing fact_sales rows that reference the old SK values.

Cell 4 — dim_customer + dim_product
# ── 2) dim_customer ───────────────────────────────────────────
dim_customer = (s
    .select("customer_id", "customer_name", "segment")
    .dropDuplicates(["customer_id"])
    .withColumn("customer_sk",
        F.row_number().over(Window.orderBy("customer_id")))
    .withColumn("_ts", F.current_timestamp()))
dim_customer.write.format("delta").mode("overwrite").saveAsTable("dim_customer")
print(f"dim_customer  : {dim_customer.count()} rows")

# ── 3) dim_product ─────────────────────────────────────────────
dim_product = (s
    .select("product_id", "product_name", "category", "sub_category")
    .dropDuplicates(["product_id"])
    .withColumn("product_sk",
        F.row_number().over(Window.orderBy("product_id")))
    .withColumn("_ts", F.current_timestamp()))
dim_product.write.format("delta").mode("overwrite").saveAsTable("dim_product")
print(f"dim_product   : {dim_product.count()} rows")
row_number() vs monotonically_increasing_id(): We use row_number().over(Window.orderBy("natural_key")) for dimension tables because it generates a stable, deterministic sequence (1, 2, 3...). monotonically_increasing_id() generates large non-sequential integers that are partition-dependent and change between runs — avoid it for SKs. We reserve monotonically_increasing_id() for fact_sk in the fact table where stability is less critical.

geo = (s .select("country", "state", "city", "region", "postal_code") .dropDuplicates()) dim_geography = (geo .join( cn.select("name_common", "_____", "_____"), # only bring over 2 cols geo.country == cn._____, # join condition "_____" # join type: keep all geo rows ) .drop("_____") # remove name_common after join (avoid clash) .withColumn("geo_sk", F.row_number().over(Window.orderBy("_____", "_____", "_____"))) .withColumn("_ts", F.current_timestamp())) dim_geography.write.format("delta").mode("overwrite").saveAsTable("_____") print(f"dim_geography : {dim_geography.count()} rows")
Why LEFT JOIN and not INNER JOIN? silver_sales only has US data. silver_countries has all 245 countries. If we used INNER JOIN, we'd only keep geo rows where country matched a country name. LEFT JOIN keeps ALL geo rows — even ones without a match in silver_countries (those get null for country_code and subregion, which is acceptable). Why .drop("name_common")? After joining, the DataFrame has BOTH geo.country AND cn.name_common. They contain the same values (that was the join condition!) so name_common is redundant. Dropping it keeps the schema clean.
✅ Challenge 3 AnswerCell 5 — the cross-source join
Cell 5 — dim_geography
# ── 4) dim_geography — enriched with silver_countries ─────────
geo = (s
    .select("country", "state", "city", "region", "postal_code")
    .dropDuplicates())

dim_geography = (geo
    .join(
        cn.select("name_common", "country_code", "subregion"),
        geo.country == cn.name_common,
        "left"
    )
    .drop("name_common")
    .withColumn("geo_sk",
        F.row_number().over(Window.orderBy("country", "state", "city")))
    .withColumn("_ts", F.current_timestamp()))
dim_geography.write.format("delta").mode("overwrite").saveAsTable("dim_geography")
print(f"dim_geography : {dim_geography.count()} rows")
Instructor Note
This is the single most architecturally interesting step in the whole workshop — it's where the two independent data sources (Superstore CSV and REST Countries API) are combined for the first time. Ask students: "Why did we need to ingest the Countries API at all?" The answer: to enrich US-only sales data with country codes and subregions for future geographic analysis. Even though the Superstore dataset only has US data, this join demonstrates the pattern for multi-country datasets in production.

# ── 5) fact_sales — join all dimension SKs ────────────────────
dc = spark.table("dim_customer") .select("customer_id",  "customer_sk")
dp = spark.table("dim_product")  .select("product_id",   "product_sk")
dg = spark.table("dim_geography").select("country", "state", "city", "geo_sk")

fact_sales = (s
    .join(dc, "customer_id", "left")
    .join(dp, "product_id",  "left")
    .join(dg, ["country", "state", "city"], "left")
    .withColumn("order_date_key",
        F.date_format(F.col("order_date"), "yyyyMMdd").cast(IntegerType()))
    .withColumn("ship_date_key",
        F.date_format(F.col("ship_date"),  "yyyyMMdd").cast(IntegerType()))
    .withColumn("fact_sk", F.monotonically_increasing_id())
    .select(
        "fact_sk",    "order_id",
        "order_date_key", "ship_date_key",
        "customer_sk", "product_sk", "geo_sk",
        "sales", "quantity", "discount", "profit",
        "days_to_ship", "profit_margin", "net_revenue",
        "_ingestion_ts"))
fact_sales.write.format("delta").mode("overwrite").saveAsTable("fact_sales")
print(f"fact_sales    : {fact_sales.count()} rows")

# ── Verify all 5 Gold tables exist ────────────────────────────
spark.sql("SHOW TABLES").filter(
    "tableName LIKE 'dim_%' OR tableName LIKE 'fact_%'"
).show()
Why load dim tables again with .select()? We only need the SK column from each dim for the join. Loading only ["customer_id","customer_sk"] instead of all 4 customer columns reduces shuffle data during the join — especially important with large dimension tables in production.

Why multi-column join for dim_geography? There is no single column that uniquely identifies a geo combination — we need all three of country + state + city together as a composite key. The list syntax ["country","state","city"] tells Spark to match on all three simultaneously.

Query 1 — Sales by product category (tests dim_product join)
SQL — paste into SQL endpoint query editor
SELECT
    p.category,
    ROUND(SUM(f.sales),   0) AS total_sales,
    ROUND(SUM(f.profit),  0) AS total_profit,
    ROUND(AVG(f.profit_margin), 1) AS avg_margin_pct
FROM fact_sales f
JOIN dim_product p ON f.product_sk = p.product_sk
GROUP BY p.category
ORDER BY total_sales DESC;
✅ Expected order: Technology (~$836K) > Furniture (~$742K) > Office Supplies (~$719K)
Query 2 — Revenue by year (tests dim_date join)
SQL
SELECT
    d.year,
    COUNT(DISTINCT f.order_id) AS total_orders,
    ROUND(SUM(f.sales), 0)     AS revenue,
    ROUND(AVG(f.days_to_ship), 1) AS avg_days_to_ship
FROM fact_sales f
JOIN dim_date d ON f.order_date_key = d.date_key
GROUP BY d.year
ORDER BY d.year;
✅ Expected: revenue grows each year 2014→2017 (growth trend validates date join)
Query 3 — Top 5 customers (tests dim_customer join)
SQL
SELECT
    c.customer_name,
    c.segment,
    COUNT(DISTINCT f.order_id)     AS orders,
    ROUND(SUM(f.sales), 0)         AS total_sales,
    ROUND(AVG(f.profit_margin), 1) AS avg_margin_pct
FROM fact_sales f
JOIN dim_customer c ON f.customer_sk = c.customer_sk
GROUP BY c.customer_name, c.segment
ORDER BY total_sales DESC
LIMIT 5;

These 3 queries are the proof that the star schema works. If any join returns nulls or 0 revenue, there's a bug to investigate. The SQL endpoint responds instantly — there's no Spark startup delay because it reads Delta files directly via a SQL engine.
