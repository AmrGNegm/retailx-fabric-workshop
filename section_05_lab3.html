# ── Add derived business columns ──────────────────────────────
df = (df
    .withColumn("order_year",    F.year(F.col("order_date")))
    .withColumn("order_month",   F.month(F.col("order_date")))
    .withColumn("order_quarter", F.quarter(F.col("order_date")))
    .withColumn("days_to_ship",  F.datediff(F.col("ship_date"), F.col("order_date")))

    # Safe profit margin — handles edge case of sales = 0
    .withColumn("profit_margin", F.round(
        F.when(F.col("sales") != 0, F.col("profit") / F.col("sales") * 100)
         .otherwise(F.lit(0.0)), 2))

    .withColumn("net_revenue",   F.round(F.col("sales") * (1 - F.col("discount")), 2))
    .dropDuplicates(["order_id", "product_id"])
    .filter(F.col("order_date").isNotNull())
    .withColumn("_ingestion_ts", F.current_timestamp()))

print(f"Transformed: {df.count()} rows, {len(df.columns)} columns")
display(df.select("order_id","order_date","days_to_ship",
                  "profit_margin","net_revenue","order_year","order_quarter").limit(5))
Why F.when() for profit_margin? In PySpark, dividing a double by zero gives Infinity or NaN — not an error, but those values cause problems in Power BI aggregations. F.when(sales != 0, ...) returns null instead of Infinity when sales is 0, which Power BI handles cleanly as a blank.

Why datediff(ship_date, order_date)? The argument order matters: datediff(end, start) = end − start. If reversed, you get negative days_to_ship which is nonsensical.
Instructor Note
The F.when() safe division pattern is one of the most important patterns in production PySpark. Ask students: "What does Infinity look like in a Power BI card visual?" It breaks the visual entirely. The safe pattern costs nothing extra in performance and prevents an entire class of downstream bugs. Also note: datediff argument order is a very common mistake — always test with a known date pair to verify positive values.

💡 Hint — F.coalesce() pattern
F.coalesce(col1, col2, ...) returns the first non-null value in the list. Use it to fill nulls with a literal default: F.coalesce(F.trim(F.col("subregion")), F.lit("Unknown")) # If subregion is null OR empty after trim → returns "Unknown" F.coalesce(F.col("capital"), F.lit("N/A")) # If capital is null → returns "N/A" However: coalesce only handles NULL, not empty string "". If you want to handle both null AND "" → wrap with F.nullif() or use F.when(): F.when(F.col("subregion").isNull() | (F.col("subregion") == ""), F.lit("Unknown")) .otherwise(F.col("subregion"))
df = (df .filter(F.col("name_common") != "") .filter(F.col("country_code") != "") .withColumn("name_common", F.trim(F.col("_____"))) .withColumn("region", F.trim(F.col("_____"))) .withColumn("subregion", F.coalesce(F.trim(F.col("subregion")), F.lit("_____"))) .withColumn("capital", F.coalesce(F.col("capital"), F.lit("_____"))) .withColumn("currency_code", F.coalesce(F.col("currency_code"), F.lit("_____"))) .withColumn("population_m", F.round(F.col("population") / _____, 3)) .dropDuplicates(["_____"]) .withColumn("_ingestion_ts", F.current_timestamp()))

NB-02-SILVER-Countries — complete notebook code
# Parameters
p_FullLoad = True
SRC_TABLE  = "raw_countries"
DEST_TABLE = "silver_countries"

from pyspark.sql import functions as F

# ── Read from Bronze ──────────────────────────────────────────
df = spark.table(SRC_TABLE)

# ── Clean and standardise ─────────────────────────────────────
df = (df
    .filter(F.col("name_common")  != "")
    .filter(F.col("country_code") != "")
    .withColumn("name_common",   F.trim(F.col("name_common")))
    .withColumn("region",        F.trim(F.col("region")))
    .withColumn("subregion",     F.coalesce(F.trim(F.col("subregion")),  F.lit("Unknown")))
    .withColumn("capital",       F.coalesce(F.col("capital"),            F.lit("N/A")))
    .withColumn("currency_code", F.coalesce(F.col("currency_code"),      F.lit("Unknown")))
    .withColumn("population_m",  F.round(F.col("population") / 1e6, 3))
    .dropDuplicates(["country_code"])
    .withColumn("_ingestion_ts", F.current_timestamp()))

# ── DQ Checks ─────────────────────────────────────────────────
total = df.count()
assert df.filter(F.col("country_code").isNull()).count() == 0, "DQ: null country_code"
assert df.filter(F.col("name_common") == "").count()     == 0, "DQ: empty name_common"
print(f"[DQ] {total} countries — all checks passed")

# ── Write to Silver ────────────────────────────────────────────
(df.write
   .format("delta")
   .mode("overwrite")
   .option("overwriteSchema", "true")
   .saveAsTable(DEST_TABLE))

print(f"Written {DEST_TABLE}: {total} rows")
display(spark.table(DEST_TABLE)
        .select("country_code","name_common","region","subregion",
                "capital","population_m","currency_code")
        .limit(5))
Why filter empty strings AND use coalesce? The filter at the top removes rows where critical keys are empty — these records aren't useful at all. The coalesce inside handles optional/nullable fields that we want to keep but fill with a default. Two different problems, two different solutions.
Instructor Note
F.coalesce() is one of the most-used functions in production data pipelines. Ask students: "What is the difference between null and an empty string in Spark?" The answer: null means "no value at all", empty string means "a value that is blank". coalesce handles null; an explicit filter or F.when handles empty string. Both need to be handled — that's why this notebook has both a .filter() and .withColumn(coalesce) pattern.

💡 Hint — Dependency types
Fabric pipelines have 4 dependency types: → ✅ On success — next activity runs only if previous succeeded → ❌ On failure — next activity runs only if previous FAILED (useful for alerts) → ✔️ On completion — next activity always runs regardless of outcome → ⚠️ On skip — runs if the previous activity was skipped For this project: always use "On success" unless building failure-handling branches. The visual result should look like this: [BRONZE-Countries] ──success──→ [SILVER-Sales] └─success──→ [SILVER-Countries] Both Silver activities start simultaneously as soon as Bronze completes successfully. This saves ~1–2 minutes per pipeline run vs running them sequentially.
Instructor Note
The fork pattern (one → many) is the most important pipeline topology to understand. When BRONZE-Countries succeeds, the pipeline engine evaluates all outgoing dependencies simultaneously. SILVER-Sales and SILVER-Countries both become eligible to run and are dispatched in parallel to available Spark clusters. There is no extra configuration for this — the parallel execution is implicit from the topology.
