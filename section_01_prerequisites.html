Microsoft Fabric Basics
Workloads, workspaces, capacities, and OneLake
Required Intermediate
▼

Microsoft Fabric is a unified analytics platform that combines data engineering, data science, data warehousing, real-time analytics, and Power BI into a single SaaS product — all sharing one underlying storage layer called OneLake. Unlike traditional architectures where each tool has its own storage, Fabric means a PySpark notebook and a Power BI report both read from the exact same Delta files with no data movement between them. You need to understand workspaces (where items live), capacities (compute that powers the platform), and the workload switcher (the persona-based navigation).

What you need to know for this project
  • How to create a workspace and assign it to a Fabric trial capacity
  • How to switch between Data Engineering, Power BI, and Data Warehouse workloads
  • What a Lakehouse is and the difference between Tables and Files sections
  • That OneLake means one copy of data — no imports, no movement between tools

Delta Lake & Delta Tables
The file format that powers every table in this project
Required Intermediate
▼

Delta Lake is an open-source storage layer that adds ACID transactions, schema enforcement, and time travel on top of regular Parquet files stored in a data lake. In Microsoft Fabric, every managed table in a Lakehouse is a Delta table — meaning it gets a transaction log (_delta_log) that records every change, supports merge/upsert operations, and can be queried by both Spark (PySpark) and SQL (T-SQL via the SQL Endpoint) simultaneously. You will use Delta as the write format for every layer in this project: Bronze, Silver, Gold, and Platinum.

What you need to know for this project
  • How to write a DataFrame to Delta: df.write.format("delta").saveAsTable("name")
  • Difference between mode("overwrite") and mode("append")
  • What ACID transactions mean: why Delta never leaves a table in a broken state
  • That Delta tables in Fabric are automatically available via T-SQL — no extra steps

PySpark Fundamentals
Reading, transforming, and writing DataFrames with Apache Spark
Required Advanced
▼

PySpark is the Python API for Apache Spark — a distributed computing engine that processes data in parallel across a cluster. In Fabric notebooks, a SparkSession is pre-configured for you; you just write PySpark code and Fabric handles the cluster. The key mental model is that a DataFrame is an immutable, lazily-evaluated distributed table — transformations like .withColumn() and .filter() build a query plan but don't execute until you call an action like .count() or .write(). All columns are referenced using F.col("name") from pyspark.sql.functions.

What you need to know for this project
  • Reading a table: spark.table("table_name")
  • Adding/transforming columns: df.withColumn("col", F.expr(...))
  • Filtering rows: df.filter(F.col("x").isNotNull())
  • Joining two DataFrames: df1.join(df2, "key_col", "left")
  • Window functions for row numbers: F.row_number().over(Window.orderBy("col"))
  • Writing to Delta: df.write.format("delta").mode("overwrite").saveAsTable("name")

Medallion Architecture
Why we layer data: Bronze, Silver, Gold, Platinum
Recommended Foundational
▼

The medallion architecture (also called multi-hop architecture) is a data design pattern that organizes a data lakehouse into layers of progressively increasing quality and structure. The Bronze layer holds raw, unmodified source data as an immutable archive. Silver cleans, types, and applies business rules. Gold builds analytics-ready structures like star schemas. Platinum serves consumption layers like Power BI. Each layer has a clear contract and a single direction of data flow — you never write back to a previous layer. This pattern improves reliability, debuggability, and reprocessing without data loss.

What you need to know for this project
  • Bronze is write-once, read-always — never mutate raw data
  • Silver resolves nulls, casts types, renames columns, deduplicates
  • Gold applies business logic: star schema, surrogate keys, joins
  • Platinum optimizes for the consumer: DirectLake, pre-aggregations, RLS

Star Schema & Dimensional Modeling
Fact tables, dimension tables, and surrogate keys
Recommended Intermediate
▼

A star schema is a dimensional modeling pattern where a central fact table (containing measurable business events like sales transactions) is surrounded by dimension tables (containing descriptive attributes like customer name, product category, or date). The fact table contains numeric measures (sales, quantity, profit) and foreign key references to each dimension via surrogate keys — integer keys generated by the data platform that replace natural business keys. Power BI performs best on star schemas because they minimize join complexity and allow VertiPaq (the Power BI engine) to compress dimension data aggressively.

What you need to know for this project
  • What a fact table contains: measures (sales, profit) + foreign keys (date_key, customer_sk)
  • What a dimension table contains: descriptive attributes (name, category, region)
  • Why we use surrogate keys (integers) instead of natural keys (order IDs, strings)
  • Why single-direction filtering is the default in Power BI star schemas

Power Query M Language
Used in Dataflow Gen2 to transform the Sales CSV
Recommended Beginner
▼

M (Power Query Formula Language) is the functional transformation language used inside Power BI Dataflows, Power BI Desktop, and Microsoft Fabric's Dataflow Gen2. An M query is a series of let steps where each step builds on the previous one — similar to a pipeline of transformations. In this project you use M inside Dataflow Gen2 to read the Superstore CSV from GitHub, rename all 21 columns to snake_case, and cast types — all without writing any Python or SQL. You don't need to write M from scratch; we provide the code and you need to understand what each step does.

What you need to know for this project
  • The structure of an M query: let ... in result
  • How Table.RenameColumns renames multiple columns at once
  • How Table.TransformColumnTypes sets data types
  • How to open the Advanced Editor in DFG2 and paste M code

DAX Basics
Writing measures for the Power BI semantic model
Recommended Intermediate
▼

DAX (Data Analysis Expressions) is the formula language used in Power BI, Analysis Services, and Microsoft Fabric semantic models to define measures, calculated columns, and calculation groups. Unlike Excel formulas that operate on cells, DAX measures operate on filter context — the set of filters active when a visual evaluates the measure. The most important function is CALCULATE(), which modifies filter context. In this project you write 8 measures covering aggregations, ratios, time intelligence (YoY comparison), and averages — all using the VAR / RETURN pattern for clean, readable code.

What you need to know for this project
  • The difference between a measure (dynamic, context-aware) and a calculated column (row-by-row)
  • How to use VAR ... RETURN for readable multi-step measures
  • How CALCULATE() modifies filter context
  • How DATEADD() shifts the date context for time intelligence
  • How DIVIDE(numerator, denominator, alternate) avoids division-by-zero errors

REST APIs & JSON
How to call a public API and parse the response in Python
Bonus Beginner
▼

REST (Representational State Transfer) is the architectural style behind most public web APIs. You interact with a REST API by sending HTTP requests (GET, POST, etc.) to a URL endpoint, and the server responds with data — almost always formatted as JSON (JavaScript Object Notation), a human-readable key-value structure. In Lab 2 of this project, you call the REST Countries API using Python's requests library, receive a JSON array of 250 country objects, parse each object into a Python dict with a helper function, and load the result into a PySpark DataFrame. Understanding JSON structure (nested objects, arrays, optional keys) is essential for this lab.

What you need to know for this project
  • How to make a GET request: response = requests.get(url, timeout=30)
  • How to parse JSON: data = response.json() returns a Python list or dict
  • How to safely access nested keys: obj.get("key", {}).get("subkey", "")
  • How to handle optional/missing keys using .get("key", default_value)
