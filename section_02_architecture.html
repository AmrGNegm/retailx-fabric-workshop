📁 GitHub CSV — Superstore Sales ✕ close

The Superstore dataset is hosted as a raw CSV on GitHub (Plotly's public datasets repository). It contains 9,994 rows of retail transactions from 2014 to 2017 across the United States, with 21 columns covering order details, shipping, customer info, product category, and financials (sales, profit, discount).

Ingested by: Dataflow Gen2 (DFG2-Sales-CSV) using Power Query M
Lands in: Bronze layer → raw_sales Delta table
URL: https://raw.githubusercontent.com/plotly/datasets/master/superstore.csv

🌐 REST Countries API ✕ close

The REST Countries API is a free, public, no-authentication API that returns information about every country in the world — including name, region, subregion, population, area, capital city, currency, and GPS coordinates. It returns a JSON array of ~250 country objects, each with deeply nested fields.

Ingested by: NB-01-BRONZE-Countries-API (PySpark notebook)
Raw JSON saved to: Files/bronze/countries/countries_{ts}.json
Lands in: Bronze layer → raw_countries Delta table
URL: https://restcountries.com/v3.1/all

🥉 Bronze Layer — Raw, Immutable Landing Zone ✕ close

The Bronze layer is a write-once archive of source data. No transformations, no cleaning, no business logic — just an exact copy of what arrived from the source. Raw JSON files are also saved to Files/bronze/ before parsing.

Rules
✓ Never modify after landing
✓ Append-only writes preferred
✓ Keep source column names as-is
✓ Store raw JSON files too
Tables
raw_sales — ~9,994 rows
raw_countries — ~250 rows
Format: Delta (overwrite on reload)
Audit cols: _ingestion_ts
🥈 Silver Layer — Cleaned, Typed, Business-Ready ✕ close

Silver applies standardisation and business logic to Bronze data. Column names become snake_case, types are cast correctly, nulls are resolved, duplicates are dropped, and derived metrics are computed.

Transformations
✓ Rename all columns → snake_case
✓ Cast dates, numbers, integers
✓ Derive: profit_margin, days_to_ship
✓ Derive: net_revenue, order_year/month
✓ Drop duplicates on business key
✓ Null-fill subregion, capital, currency
Tables
silver_sales — ~9,994 rows
silver_countries — ~245 rows
Format: Delta (overwrite on reload)
Tool: NB-02 (two separate notebooks)
🥇 Gold Layer — Star Schema, Surrogate Keys ✕ close

Gold builds the analytics-ready star schema. Natural keys are replaced with integer surrogate keys. Dimension tables are denormalized. The fact table holds only measures and foreign keys — no descriptive text.

Star schema
fact_sales (center)
→ dim_date (1,461 rows)
→ dim_customer (~793)
→ dim_product (~1,850)
→ dim_geography (~632)
Key patterns
✓ Surrogate keys via Window + row_number
✓ dim_geography enriched from silver_countries
✓ dim_date generated via SQL sequence
✓ All joins: left join from fact → dim
Tool: NB-03 (single notebook)
💎 Platinum Layer — Power BI DirectLake Serving ✕ close

Platinum is the consumer-facing layer. The semantic model reads Delta files directly from OneLake — no import, no scheduled refresh. Any new data written by the pipeline is instantly visible in the Power BI report on the next query.

Semantic model
Name: SM_RetailX
Mode: DirectLake
Tables: 5 (1 fact + 4 dims)
Relationships: 4 (M:1, single-dir)
DAX Measures: 8
Power BI Report
Name: RPT-RetailX-Overview
Visuals: Line chart, Bar, Map,
          Cards (×4), Table, Slicers
Zero refresh lag vs. pipeline output
💡 OneLake means one copy. All Fabric items in the same workspace read from the same underlying Delta files. A notebook writes to Gold; Power BI DirectLake reads the same files immediately — no ETL, no movement.

NB-01-BRONZE-Countries-API — Calls the REST Countries API, saves the raw JSON to Files/bronze/countries/, parses 250 country objects into a PySpark DataFrame, runs DQ assertions, and writes to raw_countries Delta table. Runtime ~3 min (includes Spark session start). ✕
NB-02-SILVER-Sales — Reads raw_sales, renames all 21 columns to snake_case, casts types, adds business columns (profit_margin, days_to_ship, net_revenue), drops duplicates, and writes to silver_sales Delta table. Runs in parallel with NB-02-SILVER-Countries. ✕
NB-02-SILVER-Countries — Reads raw_countries, trims whitespace, fills nulls, adds population_m (millions), deduplicates on country_code, and writes to silver_countries. Runs in parallel with NB-02-SILVER-Sales. ✕
NB-03-GOLD-StarSchema — Builds the complete star schema from Silver tables: dim_date (SQL sequence), dim_customer, dim_product, dim_geography (enriched with silver_countries), and fact_sales with all surrogate key joins. Waits for both Silver notebooks to succeed before starting. ✕

2.4 — Standards

Naming Conventions

Consistent naming across all Fabric items, notebooks, tables, and columns.

Item type Pattern Example Notes
Lakehouse LH_{ProjectName} LH_RetailX One Lakehouse per project domain
Dataflow Gen2 DFG2-{Source}-{Entity} DFG2-Sales-CSV DFG2 prefix, kebab-case
Notebooks NB-{seq}-{LAYER}-{Name} NB-01-BRONZE-Countries-API Seq ensures pipeline ordering is obvious
Data Pipeline PL-{ProjectName}-{Schedule} PL-RetailX-Daily PL prefix, schedule makes frequency explicit
Bronze tables raw_{entity} raw_sales, raw_countries raw_ prefix signals no transformation applied
Silver tables silver_{entity} silver_sales, silver_countries Layer name as prefix — layer is always obvious
Gold fact tables fact_{process} fact_sales Kimball convention — process name, not entity name
Gold dim tables dim_{entity} dim_date, dim_customer Kimball convention — singular entity name
Column names snake_case order_date, profit_margin All lowercase, underscores. No spaces. No camelCase.
Surrogate keys {entity}_sk customer_sk, product_sk _sk suffix — distinguishes from natural keys
Natural keys {entity}_id customer_id, product_id _id suffix — the original source system key
Audit columns _{name} _ingestion_ts, _source, _batch_id Underscore prefix — signals system-generated column
Semantic model SM_{ProjectName} SM_RetailX SM prefix — distinguishes from reports
Power BI report RPT-{ProjectName}-{Page} RPT-RetailX-Overview RPT prefix, kebab-case, page/scope suffix
⚠️ Follow these conventions from the very first lab. Inconsistent naming causes confusion in pipelines, broken references in Power BI, and makes onboarding new team members significantly harder. Convention is documentation.
