💡 Hint — How to create relationships in the editor
In the semantic model diagram: → Click and hold on fact_sales[order_date_key] → Drag and drop onto dim_date[date_key] → A relationship dialog appears — verify settings: • Cardinality: Many to one (*:1) • Cross filter direction: Single (fact filters dim) • Active: Yes → Click Apply / OK Repeat for the other 3 relationships. Instructor Note
Single-direction filtering is the correct default for star schemas. Bidirectional filtering allows a dimension to filter the fact in both directions — this can cause ambiguous filter paths, circular dependencies, and slow queries in models with multiple fact tables. Only use bidirectional when you have a specific, justified reason (e.g. a role-playing dimension or a specific many-to-many bridge table). Students who have used Power BI Desktop will recognise this dialog immediately.

🧩 DAX Challenge 2 — Ratio & average measures
DIVIDE · IF/ISBLANK · ROUND · AVERAGE · safe patterns
CODE Quiz
▼
🎯 DAX Code Challenge 2
Write 3 measures that compute ratios and averages. Each must handle edge cases — divide by zero, blank denominators, and null values.
  • →Profit Margin % — profit ÷ sales × 100, rounded to 1 dp. Use VAR for both. Return BLANK when sales is 0 or blank. Use DIVIDE() not / operator.
  • →Avg Order Value — Total Sales ÷ Total Orders. Use DIVIDE() with BLANK() as the alternate result.
  • →Avg Days to Ship — average of the days_to_ship column, rounded to 1 dp.
💡 Hint — DAX safe division patterns
-- DIVIDE(numerator, denominator, alternate_if_zero) -- Always use DIVIDE() not / for ratios — handles zero safely Profit Margin % = VAR vSales = [_____] VAR vProfit = [_____] RETURN IF( ISBLANK(vSales) || vSales = 0, BLANK(), ROUND(DIVIDE(_____, _____) * 100, 1) ) Avg Order Value = DIVIDE( [_____], -- numerator [_____], -- denominator _____() -- alternate when denominator = 0 ) Avg Days to Ship = ROUND(AVERAGE(fact_sales[_____]), 1)
Why DIVIDE() instead of the / operator? [Total Sales] / [Total Orders] → ERROR when Total Orders = 0 DIVIDE([Total Sales], [Total Orders], BLANK()) → returns BLANK safely Why check ISBLANK AND = 0 in Profit Margin %? When no rows match the filter context, [Total Sales] is BLANK (not 0). BLANK() / anything = BLANK() in DAX — but being explicit is clearer. 0 / anything = 0, which would show 0% instead of blank — wrong for empty slices.
✅ DAX Challenge 2 — 3 measuresAdd one at a time
DAX — Ratio & average measures
-- ── Measure 5 ────────────────────────────────────────────────
Profit Margin % =
VAR vSales  = [Total Sales]
VAR vProfit = [Total Profit]
RETURN
    IF(
        ISBLANK(vSales) || vSales = 0,
        BLANK(),
        ROUND(DIVIDE(vProfit, vSales) * 100, 1)
    )

-- ── Measure 6 ────────────────────────────────────────────────
Avg Order Value =
DIVIDE(
    [Total Sales],
    [Total Orders],
    BLANK()
)

-- ── Measure 7 ────────────────────────────────────────────────
Avg Days to Ship = ROUND(AVERAGE(fact_sales[days_to_ship]), 1)
Why reference [Total Sales] not SUM(fact_sales[sales])? Reusing measures is the correct pattern. If the definition of "Total Sales" changes later (e.g., adding a currency conversion), every measure that references it updates automatically. If you hardcode SUM(fact_sales[sales]) everywhere, you have to find and update each one manually.
Instructor Note
The DIVIDE() vs / distinction is one of the most important DAX habits to build. A report with / operators will crash or show errors when a slicer produces empty context. DIVIDE() with a BLANK() fallback shows nothing (blank cell) instead of an error — which is the correct UX. Ask students: "Which is better in a KPI card — showing an error or showing nothing?" The answer is always nothing.

🧩 DAX Challenge 3 — Sales YoY % (time intelligence)
CALCULATE · DATEADD · VAR/RETURN · ISBLANK guard
CODE Quiz
▼
🎯 DAX Code Challenge 3 — The hardest DAX measure
Write Sales YoY % — the year-over-year percentage change in sales. This requires CALCULATE with DATEADD to shift the time context back by 1 year.
  • →Store current context sales in vCurr
  • →Compute prior year sales using CALCULATE([Total Sales], DATEADD(dim_date[date], -1, YEAR)) → store in vPrev
  • →Return BLANK when vPrev is blank (first year in data = no prior year)
  • →Otherwise return: ROUND(DIVIDE(vCurr - vPrev, vPrev) * 100, 1)
  • →Verify: 2014 shows BLANK, 2015–2017 show % change values
💡 Hint — CALCULATE + DATEADD pattern
Sales YoY % = VAR vCurr = [Total Sales] VAR vPrev = CALCULATE( [_____], -- the measure to shift in time DATEADD(dim_date[_____], _____, -- column, number of periods _____) -- YEAR / MONTH / QUARTER / DAY ) RETURN IF( ISBLANK(_____), -- guard: no prior year data BLANK(), ROUND( DIVIDE(vCurr - _____, _____) * 100, 1 ) )
DATEADD(dates, number_of_intervals, interval): DATEADD(dim_date[date], -1, YEAR) → shift time context back by 1 year DATEADD(dim_date[date], -1, MONTH) → shift back by 1 month DATEADD(dim_date[date], 1, YEAR) → shift forward by 1 year CALCULATE(expression, filter) modifies the filter context. Here, DATEADD creates a new date filter that shifts the current year context back by 1 year — so when the user is looking at 2017, vPrev calculates 2016 sales. Why 2014 returns BLANK: The dataset starts in 2014. dim_date goes back to 2014-01-01. DATEADD(..., -1, YEAR) for 2014 would look for 2013 dates — none exist. CALCULATE returns BLANK. ISBLANK(BLANK()) is true. We return BLANK().
✅ DAX Challenge 3 AnswerMeasure 8
DAX — Sales YoY %
-- ── Measure 8 ────────────────────────────────────────────────
Sales YoY % =
VAR vCurr = [Total Sales]
VAR vPrev =
    CALCULATE(
        [Total Sales],
        DATEADD(dim_date[date], -1, YEAR)
    )
RETURN
    IF(
        ISBLANK(vPrev),
        BLANK(),
        ROUND(DIVIDE(vCurr - vPrev, vPrev) * 100, 1)
    )
Expected values (cross-check with Superstore benchmarks):
• 2014: (blank) — no prior year
• 2015: −2.8% (slight dip vs 2014)
• 2016: +29.5% (strong growth)
• 2017: +20.4% (continued growth)

Why DATEADD needs dim_date[date] not fact_sales[order_date]? DATEADD works on the date table column. The relationship from fact_sales → dim_date propagates the filter to the fact. Using fact_sales[order_date] directly does not leverage the date table — time intelligence functions require a proper marked date table.
